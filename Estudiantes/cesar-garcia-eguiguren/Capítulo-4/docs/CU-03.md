# CU-03 — Ver resumen de empleado

## Descripción funcional

Al hacer clic sobre una fila del listado de empleados, el sistema navega a la ficha de resumen de ese empleado. Esta pantalla agrega tres dimensiones de análisis en una sola vista: carga de trabajo actual (`WorkloadService`), tareas en curso (`WIPService`) y productividad de los últimos 30 días (`ProductivityService`). El cálculo se realiza íntegramente en el backend y se devuelve como un único objeto `EmployeeSummaryResponse`.

La pantalla dispone de cuatro pestañas de tareas que se cargan **bajo demanda** al seleccionarlas, reutilizando el endpoint `GET /tasks/filter` de CU-08.

## Qué puede hacer el usuario

### Panel de KPIs

| KPI | Descripción |
|---|---|
| **Carga de trabajo** | Porcentaje de ocupación respecto a la jornada de referencia (`WORKLOAD_REFERENCE_HOURS`). Clasifica al empleado en `optimo`, `aceptable` o `sobrecargado`. |
| **WIP (Work In Progress)** | Número de tareas abiertas asignadas. Indica si el empleado tiene demasiadas tareas en paralelo. |
| **Productividad (30 días)** | Ratio `horas_planificadas / horas_reales × 100` sobre tareas cerradas en los últimos 30 días. |

### Pestañas de tareas

Cada pestaña invoca `GET /tasks/filter` con los parámetros correspondientes al seleccionarse:

| Pestaña | Filtros enviados |
|---|---|
| **Pendientes** | `employee_id`, `status=pending` |
| **Completadas** | `employee_id`, `status=completed` |
| **Asignadas** | `employee_id` (sin `status`) |
| **Responsable** | `employee_id`, `responsable=true` |

### Guardar snapshot

El botón **Guardar snapshot** guarda el estado actual del resumen del empleado en la colección `entity_snapshots` de MongoDB (CU-17), con `entity_type = "employee"` y `entity_id = employee_id`.

## Datos mostrados

| Campo | Origen |
|---|---|
| **Nombre, departamento, cargo** | `GET /employees/{id}` → `EmployeeDetail` |
| **Carga (%)** | `WorkloadService.calculate(employee_id, detailed=True)` |
| **Horas pendientes** | `wl.pending_hours` (WorkloadResponse) |
| **WIP count** | `WIPService.calculate(employee_id).wip_count` |
| **Productividad media** | `ProductivityService.calculate(...).average_productivity` |
| **Tareas completadas (30 d)** | `wl.total_completed_tasks` |

---

## Restricciones de acceso

- **Director:** puede acceder al resumen de cualquier empleado del sistema.
- **Responsable:** solo puede acceder si el `employee_id` está en `cu.employee_ids` (embebido en el JWT en el momento del login). Un intento de acceso fuera del ámbito devuelve un 403.
- La validación de ámbito se ejecuta dos veces: una en `GET /employees/{id}` (vía `EmployeeService.get_employee`) y otra en `GET /dashboards/summary/employee/{id}` (vía `DashboardService.get_employee_summary`), para impedir el acceso directo al endpoint de dashboard eludiendo la ficha de empleado.

### Capa de rutas

El frontend realiza dos peticiones independientes al cargar la ficha: primero los datos básicos del empleado y después el bloque de KPIs. Ambos endpoints aplican el guard `require_manager_or_above`.

```python
# routes/employee.py — datos básicos del empleado
@router.get("/{employee_id}", response_model=EmployeeDetail)
def get_employee(employee_id: int,
                 db: Session     = Depends(get_db),
                 cu: CurrentUser = Depends(require_manager_or_above)):
    return EmployeeService(db).get_employee(cu, employee_id)

# routes/dashboards.py — KPIs agregados (workload + WIP + productividad)
@router.get("/summary/employee/{employee_id}")
def employee_summary(employee_id: int,
                     db: Session     = Depends(get_db),
                     cu: CurrentUser = Depends(require_manager_or_above)):
    return DashboardService(db).get_employee_summary(cu, employee_id)
```

### `DashboardService` — orquestación de KPIs

El método invoca los tres servicios de métricas secuencialmente y construye un único `EmployeeSummaryResponse`. La validación de scope se repite aquí intencionalmente para proteger el endpoint de dashboard si se accede directamente sin pasar por el endpoint de empleado.

```python
# services/dashboard_service.py
def get_employee_summary(self, cu, employee_id):
    verify_employee_exists(self.db, employee_id)
    verify_employee_scope(cu, employee_id)   # segunda barrera de scope

    wl   = WorkloadService(self.db).calculate(employee_id=employee_id, detailed=True)
    wip  = WIPService(self.db).calculate(employee_id=employee_id)
    prod = ProductivityService(self.db).calculate(
        employee_id=employee_id,
        date_from=date.today() - timedelta(days=30),   # ventana deslizante de 30 días
        date_to=date.today(),
    )

    return EmployeeSummaryResponse(
        employee_id=employee_id,
        workload=wl,
        wip=wip,
        productivity_last_30_days=prod,
        quick_stats=EmployeeQuickStats(
            workload_status=wl.status,                  # "normal" | "sobrecargado" | "subcargado"
            pending_hours=wl.pending_hours,
            pending_tasks=wl.total_pending_tasks,
            completed_last_30_days=wl.total_completed_tasks,
            wip_count=wip.wip_count,
            avg_productivity=prod.average_productivity,
        ),
    )
```

### `WorkloadService` — carga de trabajo

Convierte el `employee_id` en `user_id` (clave de la tabla `project_task`) y calcula el porcentaje de ocupación sumando las horas pendientes sobre la constante `WORKLOAD_REFERENCE_HOURS`. Si el empleado no tiene `user_id` asociado, devuelve un objeto vacío en lugar de error.

```python
# services/metrics/workload.py
def _workload_status(percentage: float) -> str:
    if percentage > WORKLOAD_OVERLOADED_THRESHOLD: return "sobrecargado"
    if percentage < WORKLOAD_UNDERLOADED_THRESHOLD: return "subcargado"
    return "normal"

class WorkloadService:
    def calculate(self, employee_id: int, detailed: bool = False, root_only: bool = False):
        user_id = get_user_id_for_employee(self.db, employee_id)
        if not user_id:
            return self._empty(detailed)   # empleado sin cuenta de usuario en Odoo

        pending_rows   = get_assigned_open_tasks(self.db, user_id, root_only=root_only)
        cutoff         = date.today() - timedelta(days=COMPLETED_WINDOW_DAYS)
        completed_rows = get_assigned_closed_tasks(self.db, user_id, date_from=cutoff, root_only=root_only)

        pending_hours = sum(max(float(r.pending_hours or 0), 0) for r in pending_rows)
        workload_pct  = round((pending_hours / WORKLOAD_REFERENCE_HOURS) * 100, 2)

        result = WorkloadResponse(
            workload_percentage=workload_pct,
            pending_hours=round(pending_hours, 2),
            available_hours=float(WORKLOAD_REFERENCE_HOURS),
            status=_workload_status(workload_pct),
            total_pending_tasks=len(pending_rows),
            total_completed_tasks=len(completed_rows),
        )
        if detailed:
            # Solo se calculan las listas detalladas cuando las solicita el caller
            result.pending_tasks   = _format_pending(pending_rows)
            result.completed_tasks = _format_completed(completed_rows)
        return result
```

### `WIPService` — tareas en paralelo

Cuenta las tareas abiertas asignadas al `user_id` del empleado. La clasificación (`optimo` / `aceptable` / `sobrecargado`) usa dos umbrales configurables.

```python
# services/metrics/wip.py
def _wip_status(count):
    if count <= WIP_OPTIMAL_THRESHOLD:    return "optimo"
    if count <= WIP_ACCEPTABLE_THRESHOLD: return "aceptable"
    return "sobrecargado"

class WIPService:
    def calculate(self, employee_id: int, root_only: bool = False) -> WIPResponse:
        user_id = get_user_id_for_employee(self.db, employee_id)
        if not user_id:
            return WIPResponse(wip_count=0, status="optimo",
                               recommendation="Sin tareas asignadas")
        count = count_open_assigned_tasks(self.db, user_id, root_only)
        return WIPResponse(
            wip_count=count,
            status=_wip_status(count),
            recommendation=_recommendation(count),   # texto de consejo según el nivel
        )
```

### `ProductivityService` — productividad últimos 30 días

La fórmula `(planned / actual) × 100` se aplica sobre las tareas cerradas con imputación real. La reutilización de este servicio desde `DashboardService` (sin parámetros de proyecto) hace que la ventana temporal sea la única restricción activa.

```python
# services/metrics/productivity.py  (fragmento — lógica de cálculo)
for r in rows:
    if r.actual_hours > 0:
        pct = (r.planned_hours / r.actual_hours) * 100
        items.append(ProductivityTaskItem(productivity_pct=round(pct, 2), ...))
        total_pct += pct
        valid += 1

return ProductivityResponse(
    average_productivity=round(total_pct / valid, 2) if valid else 0,
    total_tasks=valid,
    tasks=sorted(items, key=lambda x: x.productivity_pct, reverse=True),
)
```

### Capa de repositorio — `repositories/employee.py`

La consulta base usa `outerjoin` con `Department` para que los empleados sin departamento asignado no queden excluidos del resultado.

```python
# repositories/employee.py
def _base_query(db: Session):
    return db.query(*_base_columns()).outerjoin(
        Department, Employee.department_id == Department.id
    )

def get_employee_by_id(db: Session, employee_id: int):
    return (
        _base_query(db)
        .filter(Employee.id == employee_id, Employee.active == True)
        .first()   # None si no existe o está dado de baja
    )

def get_user_id_for_employee(db: Session, employee_id: int) -> Optional[int]:
    # Los servicios de métricas necesitan user_id (res_users) en vez de employee_id
    # porque project_task.user_ids referencia res_users, no hr_employee
    row = db.query(Employee.user_id).filter(Employee.id == employee_id).first()
    return row.user_id if row else None
```
