# CU-17 — Guardar snapshot (upsert diario)

## Descripción funcional

El sistema permite guardar el estado actual de cualquier vista calculada mediante el botón **Guardar snapshot**, presente en todas las pantallas analíticas del frontend principal. La operación implementa una semántica de **upsert diario**: si ya existe una snapshot con la misma clave compuesta `(tipo, params_hash, snapshot_date)`, el documento se sobrescribe en lugar de insertar un duplicado.

Existen tres subtipos de snapshot, cada uno con su propia colección MongoDB y endpoint:

| Subtipo | Colección | Endpoint | Clave compuesta |
|---|---|---|---|
| Métrica | `metric_snapshots` | `POST /snapshots/metrics` | `(metric_name, params_hash, snapshot_date)` |
| Gráfico | `chart_snapshots` | `POST /snapshots/charts` | `(chart_name, params_hash, snapshot_date)` |
| Entidad | `entity_snapshots` | `POST /snapshots/entities` | `(entity_type, entity_id, snapshot_date)` |

## Qué hace el sistema al guardar

### Flujo del `SnapshotService` y `SnapshotRepository`

| Paso | Clase / Función | Acción |
|---|---|---|
| 1 | `SaveSnapshotButton` (frontend) | El actor pulsa el botón. El componente recoge el `tipo`, los `params` actuales y el `data` de la vista. |
| 2 | `api/snapshots.js` | Envía `POST /snapshots/{metrics\|charts\|entities}` con `Authorization: Bearer JWT`. |
| 3 | `snapshots.router` → `require_manager_or_above` | Valida el JWT. Rechaza con 403 si el rol es `empleado`. Inyecta `CurrentUser`. |
| 4 | `SnapshotService.save_{metric\|chart\|entity}(cu, ...)` | Construye el actor (`_actor(cu)`) con `user_id`, `employee_id`, `employee_name` y `role` extraídos del JWT. Delega en el repositorio. |
| 5 | `snapshot.py → _hash_params(params)` | Serializa `params` ordenados alfabéticamente y calcula `SHA-256[:16]` para obtener `params_hash`. (Solo para métricas y gráficos.) |
| 6 | `snapshot.py → _today()` | Fija `snapshot_date = date.today().isoformat()`. |
| 7 | `snapshot.py → _upsert(coll, match, payload, actor)` | Ejecuta `update_one(match, {$set: {...}, $setOnInsert: {created_at, created_by}}, upsert=True)`. Si hay inserción, `upserted_id` está presente (`created=True`). Si es actualización, hace `find_one(match, {_id: 1})` para recuperar el `_id` (`created=False`). |
| 8 | `snapshots.router` | Devuelve `200 OK` + `{id, created, snapshot_date}`. |
| 9 | `SaveSnapshotButton` | Muestra notificación «Snapshot creada» o «Snapshot actualizada» según el campo `created`. |

### Política de upsert y garantía de unicidad

El `params_hash` garantiza que dos llamadas con los mismos parámetros en el mismo día produzcan el mismo documento. Los índices únicos de MongoDB sobre la clave compuesta actúan como red de seguridad en caso de concurrencia. El campo `$setOnInsert` preserva `created_at` y `created_by` en las actualizaciones.

---

## Datos de salida

| Campo | Descripción |
|---|---|
| **id** | `ObjectId` de MongoDB serializado como string. |
| **created** | `true` si se creó un documento nuevo; `false` si se actualizó uno existente. |
| **snapshot_date** | Fecha del día en formato `YYYY-MM-DD`. |

---

## Restricciones de acceso

- **Director y Responsable:** ambos pueden guardar snapshots de cualquier vista a la que tengan acceso.
- El guard `require_manager_or_above` rechaza con 403 a los usuarios con rol `empleado`.
- No existe control de scope sobre la operación de guardado: la responsabilidad de acceder solo a datos del propio ámbito la tienen los endpoints que calculan la vista previa.

### Capa de rutas — `routes/snapshots.py`

Hay tres endpoints `POST`, uno por tipo de snapshot. Nótese que el router **no inyecta `db`** porque MongoDB se accede sin SQLAlchemy. La factory `_svc()` instancia el servicio sin parámetros.

```python
# router prefix: /snapshots
def _svc() -> SnapshotService:
    return SnapshotService()

# ── Guardar métrica analítica ────────────────────────────────────────────────
@router.post("/metrics", response_model=UpsertResult)
def create_metric(
    body: MetricSnapshotCreate,    # fields: metric_name, params, data
    cu:   CurrentUser = Depends(require_manager_or_above),
):
    return _svc().save_metric(cu, body.metric_name, body.params, body.data)
    # UpsertResult: { "id": "...", "created": true/false, "snapshot_date": "YYYY-MM-DD" }

# ── Guardar gráfico ──────────────────────────────────────────────────────────
@router.post("/charts", response_model=UpsertResult)
def create_chart(
    body: ChartSnapshotCreate,     # fields: chart_name, params, data
    cu:   CurrentUser = Depends(require_manager_or_above),
):
    return _svc().save_chart(cu, body.chart_name, body.params, body.data)

# ── Guardar entidad (proyecto, empleado, departamento, tarea) ────────────────
@router.post("/entities", response_model=UpsertResult)
def create_entity(
    body: EntitySnapshotCreate,    # fields: entity_type, entity_id, data
    cu:   CurrentUser = Depends(require_manager_or_above),
):
    return _svc().save_entity(cu, body.entity_type, body.entity_id, body.data)
```

### Capa de servicio — `services/snapshot.py`

`_actor` construye el objeto de auditoría desde el JWT. Cada método de guardado delega en el repositorio y devuelve el resultado en el mismo formato. `save_entity` añade una validación de `entity_type` antes de persistir.

```python
def _actor(cu: CurrentUser) -> dict:
    # Almacenado en created_by / updated_by del documento MongoDB
    return {
        "user_id":       cu.user_id,
        "employee_id":   cu.employee_id,
        "employee_name": cu.employee_name,
        "role":          cu.role,
    }

class SnapshotService:

    def save_metric(self, cu, metric_name, params, data):
        oid, created, snap_date = repo.upsert_metric_snapshot(
            metric_name, params, data, _actor(cu)
        )
        return {"id": oid, "created": created, "snapshot_date": snap_date}

    def save_chart(self, cu, chart_name, params, data):
        oid, created, snap_date = repo.upsert_chart_snapshot(
            chart_name, params, data, _actor(cu)
        )
        return {"id": oid, "created": created, "snapshot_date": snap_date}

    def save_entity(self, cu, entity_type, entity_id, data):
        # Validación de dominio antes de persistir
        if entity_type not in {"project", "employee", "department", "task"}:
            raise HTTPException(
                status_code=422,
                detail="entity_type debe ser 'project' | 'employee' | 'department' | 'task'",
            )
        oid, created, snap_date = repo.upsert_entity_snapshot(
            entity_type, entity_id, data, _actor(cu)
        )
        return {"id": oid, "created": created, "snapshot_date": snap_date}
```

### Capa de repositorio — `repositories/snapshot.py`

Los tres tipos de snapshot comparten el mismo mecanismo `_upsert`. La diferencia está en la **clave compuesta** del `match` y la colección MongoDB de destino:

| Tipo | Colección | Clave de unicidad |
|---|---|---|
| Métrica | `metrics` | `metric_name` + `params_hash` + `snapshot_date` |
| Gráfico | `charts` | `chart_name` + `params_hash` + `snapshot_date` |
| Entidad | `entities` | `entity_type` + `entity_id` + `snapshot_date` |

```python
def _hash_params(params: dict) -> str:
    # Hash determinista: sort_keys asegura que el orden de las claves no importa
    normalized = json.dumps(params or {}, sort_keys=True, default=str)
    return hashlib.sha256(normalized.encode()).hexdigest()[:16]

def _today() -> str:
    return date.today().isoformat()   # "2025-05-18"

def _upsert(coll: Collection, match: dict, payload: dict, actor: dict) -> tuple[str, bool]:
    now = _now()
    update = {
        "$set":         {**payload, "updated_at": now, "updated_by": actor},
        "$setOnInsert": {"created_at": now, "created_by": actor},  # solo en inserción
    }
    result = coll.update_one(match, update, upsert=True)
    if result.upserted_id is not None:
        return str(result.upserted_id), True      # documento nuevo
    doc = coll.find_one(match, {"_id": 1})
    return str(doc["_id"]), False                  # documento actualizado

# ── Funciones públicas ────────────────────────────────────────────────────────

def upsert_metric_snapshot(metric_name, params, data, actor) -> tuple[str, bool, str]:
    snap_date   = _today()
    params_hash = _hash_params(params)
    match   = {"metric_name": metric_name, "params_hash": params_hash, "snapshot_date": snap_date}
    payload = {"metric_name": metric_name, "params": params,
               "params_hash": params_hash, "snapshot_date": snap_date, "data": data}
    oid, created = _upsert(get_metrics_collection(), match, payload, actor)
    return oid, created, snap_date

def upsert_chart_snapshot(chart_name, params, data, actor) -> tuple[str, bool, str]:
    snap_date   = _today()
    params_hash = _hash_params(params)
    match   = {"chart_name": chart_name, "params_hash": params_hash, "snapshot_date": snap_date}
    payload = {"chart_name": chart_name, "params": params,
               "params_hash": params_hash, "snapshot_date": snap_date, "data": data}
    oid, created = _upsert(get_charts_collection(), match, payload, actor)
    return oid, created, snap_date

def upsert_entity_snapshot(entity_type, entity_id, data, actor) -> tuple[str, bool, str]:
    snap_date = _today()
    # Las entidades no usan hash de params; se identifican por tipo + ID + fecha
    match   = {"entity_type": entity_type, "entity_id": entity_id, "snapshot_date": snap_date}
    payload = {"entity_type": entity_type, "entity_id": entity_id,
               "snapshot_date": snap_date, "data": data}
    oid, created = _upsert(get_entities_collection(), match, payload, actor)
    return oid, created, snap_date
```
