# CU-19 — Consultar detalle de snapshot

## Descripción funcional

El visor de snapshots (frontend en puerto 3001) permite consultar el estado exacto que tenía una vista analítica en el momento en que fue guardada. El sistema reconstruye la visualización en pantalla únicamente a partir del JSON almacenado en MongoDB, sin ejecutar ninguna consulta sobre el ERP.

El actor accede a esta pantalla seleccionando una fila del listado de snapshots (CU-18). La URL contiene `{type}` e `{id}`, que determinan qué colección consultar y qué documento recuperar.

## Qué puede ver el usuario

### Metadatos de la snapshot

| Campo | Descripción |
|---|---|
| **Tipo** | `metric`, `chart` o `entity`. |
| **Nombre** | `metric_name`, `chart_name` o `entity_type + entity_id` según el subtipo. |
| **Fecha** | `snapshot_date`: día en que fue guardada. |
| **Hash de parámetros** | `params_hash` (solo métricas y gráficos). Identifica unívocamente los parámetros usados. |
| **Parámetros** | Objeto `params` original; visible en panel expandible. |
| **Creado por** | `created_by.employee_name` + rol + fecha `created_at`. |
| **Actualizado por** | `updated_by.employee_name` + rol + fecha `updated_at` (si hubo upsert posterior). |

### Visualización reconstruida

El componente `SnapshotDetail.jsx` selecciona el renderer según el subtipo:

| Subtipo | Renderer | Qué muestra |
|---|---|---|
| `metric` | `MetricVisualizer` | Gauge, KPIs y series temporales desde `data`. |
| `chart` | `ChartVisualizer` | Gráfico interactivo (barras, tarta, líneas) desde `data`. |
| `entity` | `EntityVisualizer` | Ficha de entidad con avatar, campos y barra de progreso. |

### Eliminación (CU-20)

El botón **Eliminar** dispara `DELETE /snapshots/{type}/{id}` y redirige al listado si tiene éxito.

---

## Endpoints implicados

| Subtipo | Endpoint |
|---|---|
| Métrica | `GET /snapshots/metrics/{id}` |
| Gráfico | `GET /snapshots/charts/{id}` |
| Entidad | `GET /snapshots/entities/{id}` |

El repositorio ejecuta `find_one({_id: ObjectId(id)})`. Si no existe, responde `404 Not Found`.

---

## Decisión de diseño clave

El visor no llama a ningún endpoint del ERP. Los renderers operan exclusivamente sobre el campo `data` almacenado. Esto garantiza que la visualización sea estable aunque la lógica del frontend principal evolucione.

---

## Restricciones de acceso

- **Director y Responsable:** pueden consultar el detalle de cualquier snapshot listada.
- No se aplica filtrado por scope en la lectura. Las snapshots de rentabilidad solo pueden ser generadas por el Director, lo que limita implícitamente su exposición.
- El guard `require_manager_or_above` bloquea el acceso a usuarios con rol `empleado`.

### Capa de rutas — `routes/snapshots.py`

El listado y el detalle coexisten en el mismo router. La paginación se construye en la propia ruta con la función helper `_paginated`, ya que el servicio devuelve `(items, total)` directamente.

```python
# router prefix: /snapshots

def _paginated(items, total, page, page_size) -> dict:
    return {
        "items": items, "total": total, "page": page, "page_size": page_size,
        "total_pages": max(1, math.ceil(total / page_size)) if total > 0 else 1,
    }

# ── Listado paginado de snapshots de métrica ─────────────────────────────────
@router.get("/metrics")
def list_metrics(
    metric_name: Optional[str] = Query(None),
    date_from:   Optional[str] = Query(None),
    date_to:     Optional[str] = Query(None),
    page:        int           = Query(1, ge=1),
    page_size:   int           = Query(50, ge=1, le=200),
    _cu: CurrentUser = Depends(require_manager_or_above),
):
    skip = (page - 1) * page_size
    items, total = _svc().list_metrics_raw(metric_name, date_from, date_to,
                                            limit=page_size, skip=skip)
    return _paginated(items, total, page, page_size)

# ── Detalle de una snapshot concreta ─────────────────────────────────────────
@router.get("/metrics/{snapshot_id}")
def get_metric(snapshot_id: str, _cu: CurrentUser = Depends(require_manager_or_above)):
    return _svc().get_metric(snapshot_id)   # 404 si no existe

# Análogos para charts y entities:
# GET /snapshots/charts            → list_charts
# GET /snapshots/charts/{id}       → get_chart
# GET /snapshots/entities          → list_entities
# GET /snapshots/entities/{id}     → get_entity
```

### Capa de servicio — `services/snapshot.py`

Cada método de lectura eleva un `404` si el repositorio devuelve `None`. `list_metrics_raw` devuelve la tupla `(items, total)` directamente para que sea la ruta quien construya la respuesta paginada.

```python
class SnapshotService:

    def get_metric(self, id_str: str):
        doc = repo.get_metric_snapshot(id_str)
        if not doc:
            raise HTTPException(status_code=404, detail="Snapshot no encontrada")
        return doc   # dict con _id serializado como "id" (string)

    def list_metrics_raw(self, metric_name=None, date_from=None, date_to=None,
                         limit=50, skip=0):
        # Delega en el repositorio; devuelve (items: list[dict], total: int)
        return repo.list_metric_snapshots(metric_name, date_from, date_to, limit, skip)

    def get_chart(self, id_str: str):
        doc = repo.get_chart_snapshot(id_str)
        if not doc:
            raise HTTPException(status_code=404, detail="Snapshot no encontrada")
        return doc

    def get_entity(self, id_str: str):
        doc = repo.get_entity_snapshot(id_str)
        if not doc:
            raise HTTPException(status_code=404, detail="Snapshot no encontrada")
        return doc
```

### Capa de repositorio — `repositories/snapshot.py`

`_oid` convierte el string recibido en `ObjectId` de BSON. `_serialize` mapea `_id → id` para que el frontend reciba JSON estándar. `list_metric_snapshots` ejecuta dos operaciones MongoDB: `find` con `limit/skip` y `count_documents` sobre el mismo filtro.

```python
def _oid(id_str: str) -> ObjectId:
    try:
        return ObjectId(id_str)
    except (InvalidId, TypeError) as exc:
        raise ValueError(f"ID inválido: {id_str}") from exc

def _serialize(doc: dict | None) -> dict | None:
    if doc is None:
        return None
    doc = dict(doc)
    doc["id"] = str(doc.pop("_id"))   # ObjectId → string
    return doc

def get_metric_snapshot(id_str: str) -> dict | None:
    return _serialize(
        get_metrics_collection().find_one({"_id": _oid(id_str)})
    )

def list_metric_snapshots(metric_name=None, date_from=None, date_to=None,
                           limit=50, skip=0) -> tuple[list, int]:
    filt = {}
    if metric_name: filt["metric_name"] = metric_name
    if date_from:   filt["snapshot_date"] = {"$gte": date_from}
    if date_to:
        filt.setdefault("snapshot_date", {})["$lte"] = date_to
    coll  = get_metrics_collection()
    total = coll.count_documents(filt)
    docs  = list(coll.find(filt).sort("snapshot_date", -1).skip(skip).limit(limit))
    return [_serialize(d) for d in docs], total

# Análogos: get_chart_snapshot / list_chart_snapshots
#           get_entity_snapshot / list_entity_snapshots
```
