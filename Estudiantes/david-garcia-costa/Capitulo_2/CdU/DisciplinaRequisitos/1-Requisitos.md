# Requisitos funcionales

## RF01. Autenticacion

El sistema debe permitir iniciar y cerrar sesion.

## RF02. Introduccion de documentacion funcional

El sistema debe permitir introducir documentacion funcional en formato texto libre, DRF, DDS o combinacion de ellos.

## RF03. Asociacion de documentacion a proyecto

El sistema debe permitir asociar la documentacion introducida a un proyecto cuando dicha asociacion no este disponible inicialmente.

## RF04. Consulta de documentacion registrada

El sistema debe permitir consultar las referencias de la documentacion funcional asociada a un proyecto.

## RF05. Gestion de casos de uso locales

El sistema debe permitir extraer, listar, consultar, crear, actualizar y eliminar casos de uso locales.

## RF06. Gestion de requisitos funcionales locales

El sistema debe permitir extraer, listar, consultar, crear, actualizar y eliminar requisitos funcionales locales.

## RF07. Generacion de escenarios Gherkin

El sistema debe permitir generar y listar escenarios Gherkin a partir de casos de uso y requisitos funcionales.

## RF08. Gestion de borradores

El sistema debe permitir crear, listar, consultar, enriquecer con feedback y rechazar borradores de caso de prueba.

## RF09. Aceptacion y publicacion de casos de prueba

El sistema debe permitir aceptar un borrador valido, transformarlo en un caso de prueba final y publicarlo en Kiwi TCMS.

## RF10. Consulta en Kiwi TCMS

El sistema debe permitir buscar y ver casos de prueba en Kiwi TCMS.

## RF11. Gestion de sesiones de trabajo

El sistema debe permitir seleccionar sesiones existentes y crear nuevas sesiones para continuar o iniciar el trabajo en un contexto diferenciado.

## RF12. Almacenamiento de resultados

El sistema debe permitir guardar los resultados generados o editados durante una sesion, incluyendo casos de uso, requisitos funcionales y borradores.

## RF13. Trazabilidad funcional

El sistema debe mantener trazabilidad entre proyecto, sesion, documentacion, casos de uso, requisitos funcionales, escenarios Gherkin, borradores y casos de prueba publicados.

---

# Requisitos no funcionales

## RNF01. Consistencia

El sistema debe mantener coherencia entre los artefactos generados y gestionados durante todo el flujo funcional.

## RNF02. Trazabilidad

El sistema debe conservar referencias de origen y relaciones entre artefactos funcionales y de prueba.

## RNF03. Usabilidad

El sistema debe ofrecer operaciones claras, diferenciadas y comprensibles.

## RNF04. Integracion

El sistema debe integrarse con Kiwi TCMS para registrar, buscar y consultar casos de prueba.

## RNF05. Mantenibilidad

El sistema debe organizar sus responsabilidades funcionales de manera modular.

## RNF06. Extensibilidad

El sistema debe permitir incorporar nuevas fuentes y nuevas operaciones sin alterar la estructura funcional general.

## RNF07. Identificacion de contexto

El sistema debe preservar la relacion entre cada artefacto y su proyecto de referencia.
